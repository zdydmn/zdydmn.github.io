1.初始化项目 ​

bash
npm init
npm i -D webpack webpack-cli

创建入口文件/src/index.js ，写入初始代码进行打包测试。

javascript
console.log('Hello webpack!')

在package.json中配置指令

json
"scripts": {
    "build": "webpack ./src/index.js"
  },

执行npm run build生成dist文件夹，并且内部含有main.js说明已经打包成功了。

2. 修改执行语句 ​

webpack配置文档：中文官网

在根目录下创建webpack.config.js文件，实现webpack更加丰富的自定义配置

修改package.json中配置的指令，使打包时按照自定义配置进行打包，而非默认配置。

json
"scripts": {
    "build": "webpack --config webpack.config.js"
  },

3. webpack.config.js基础配置 ​

javascript
// webpack.config.js

const path = require("path");

module.exports = {
  mode: "development",
  entry: path.resolve(__dirname, "./src/index.js"),
  output: {
    filename: 'index.js',
    path: path.resolve(__dirname,'../dist')
  },
  resolve: {
    alias: {
      "@": path.resolve(__dirname, "./src/")
    },
    extensions: [".js", ".jsx"],
    modules: [
      path.join(__dirname, './node_modules'),
    ]
  },
  module: {
    rules: [
      {
        test: /\.css$/i,
        use: ['css-loader']
      }
    ]
  },
  plugins: [
    new CleanWebpackPlugin(),
    new HtmlWebpackPlugin({
      template: path.resolve(__dirname, "template.html")
    })
  ],
  devServer: {
    port: 8080,
    historyApiFallback: true,
    static: {
      directory: path.join(__dirname, "public")
    }
  }
}

mode（指定模式） ​

mode： 指定webpack的构建模式，控制webpack的默认行为

  • development： 开发环境模式
  • production：生产环境模式

生产环境相较于开发环境，增加了tree-shaking，moduleId等优化行为

entry（指定入口） ​

entry： 指定webpack的入口文件。可以是一个路径字符串或一个对象等。如果是一个对象，则会对多个依赖关系进行分析

javascript
const path = require("path");

// 单入口
module.exports = {
  entry: path.resolve(__dirname, "./src/index.js"),
}


// 多入口
module.exports = {
  entry: {
    app: path.resolve(__dirname, '../src/index.js'),
    app2: path.resolve(__dirname, '../src/index2.js')
  }
}

output（指定出口） ​

output：指定webpack的输出配置。包括输出目录、输出chunk文件名，entry文件名publicPath等

静态资源最终访问路径 = output.publicPath + 资源loader或插件等配置路径

javascript
module.exports = {
  output: {
    path: path.join(__dirname, '../dist'), // 输出目录
    filename: 'js/[name].[chunkhash].js', // 输出文件名
    //在webpack打包遇到import时会把他当如chunk文件下
    // 配置chunkFilename可以减小入口体积，减少首次加载时间。只有遇到import才会去加载对应资源
    chunkFilename: "chunk/[name].[chunkhash].js",
    publicPath: "/" // 静态资源的公共url, 默认为空
  },
};

chunkhash：chunkhash就是模块的hash，根据不同的入口文件进行依赖关系的解析，构建对应的Chunk，每个Chunk都有自己的hash值。

resolve（指定模块查找配置） ​

resolve：指定webpack查找模块时的配置，比如指定别名、npm包相关查找规则

javascript
module.exports = {
 resolve: {
   symlinks: true,
   // alias:设置模块别名，简化引入
   alias: {
     "@": path.resolve(__dirname, "./src/") // 使用@来代替src目录
   },
   // extensions:设置那些文件引入可以省略后缀名
   extensions: [".js", ".jsx"],
   // modules:设置webpack解析模块时的搜索目录
   modules: [
     path.join(__dirname, './node_modules'),
   ]
 }
}

module（模块转化规则） ​

loader: Loader可以看作具有文件转换功能的翻译员，可以将非js模块转化为webpack可以识别的js模块。loader会讲所有类型的文件转化为应用程序的依赖图(可以直接引用的模块)；

module: webpack默认是不能处理所有静态资源的，所以我们需要通过loader来帮助我们，将webpack不能识别的资源转化成webpack可以识别的内容,module指定loader、parse等配置

javascript
module.exports = {
  module: {
    rules: [
      {
        test: /\.css$/i, //正则匹配.css结尾的文件夹
        use: ['css-loader'] // 因为webpack只能识别js，借助css-loader解析css文件
      }
    ]
  }
}

plugins（插件） ​

plugin：Plugin是以事件的形式工作，webpack会在运行的各个阶段暴露出对应的事件，Plugin会监听对应的事件去执行相应的逻辑处理

plugins：自定义webpack的构建过程，增强webpack能力

javascript
module.exports = {
  plugins: [
    new CleanWebpackPlugin(), //情况目录
    new HtmlWebpackPlugin({
      // 指定打包后html文件模版
      template: path.resolve(__dirname, "template.html")
    })
  ]
}

devServer（开发服务器） ​

配合 webpack-dev-server插件构建开发服务器。使用前需要安装 webpack-dev-server

bash
npm add webpack-dev-server -D
javascript
module.exports = {
    devServer: {
        port: 8080, //启动端口
        historyApiFallback: true, // 支持history模式
        static: {
            directory: path.join(__dirname, "public")//编译后需要一些静态资源，引入静态资源地址
        }
    }
}
